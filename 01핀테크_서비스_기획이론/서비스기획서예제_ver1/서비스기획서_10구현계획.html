구현 계획(Implementation Roadmap) 이론

구현 계획은 서비스 개발을 단계적으로 실행하기 위한 실질적인 전략 문서이다. 우선순위, 일정, 기술 스택, 인프라 구성을 명확히 하여 MVP 개발과 운영을 리스크를 통제 가능한 범위 내에서 진행할 수 있도록 돕는다.

  • MVP 범위 · 개발 우선순위 정의
  • 기술 스택 및 인프라(컨테이너, DB, 배포 구조) 명시
  • 단계별 마일스톤 설정을 통한 일정 관리 및 리스크 최소화
  • 향후 고도화(버전업)와 AI 기능 확장의 기준 제공

9. 구현 계획

9-1. 기술 스택 (MVP 기준)

  • 프론트엔드
    - React 기반 화면을 HTML로 구성, Nginx 컨테이너에서 정적 파일로 서빙
    - 파일 구조: index.html, styles.css, 와이어프레임별 wf_*.html
  • 백엔드
    - Python FastAPI + Gunicorn(UvicornWorker)
    - 컨테이너: python:3.10-slim 기반, REST API 제공
    - 실행: gunicorn main:app -k uvicorn.workers.UvicornWorker --bind 0.0.0.0:8000 --workers 1
  • 데이터베이스(DB)
    - MySQL 8.0 (컨테이너명: easyfin-mysql)
    - DB명: easyfin_db, 계정: easyfin / easyfin
    - Docker 볼륨: mysql_data:/var/lib/mysql 를 통해 데이터 영속화
  • 인프라 · 배포
    - docker-compose 기반 3티어 구조 (frontend / backend / db)
    - 서비스 간 통신: docker 브리지 네트워크, 서비스명 기반 DNS(db:3306 등)
    - 재시작 정책: restart: unless-stopped (장애 시 자동 복구 목적)
  • AI / 향후 확장
    - 초기: 더미 데이터 기반 AI 브리핑/요약 영역만 와이어프레임에 노출
    - 이후: 외부 LLM API 또는 자체 RAG 서버 연동(별도 서비스로 분리) 계획

9-2. 단계별 구현 로드맵

  1. 1단계 – 기획 · 설계 정리
    - 서비스 목적·목표, 페르소나, 문제 정의 문서 확정
    - IA 정보구조도, 고객 여정맵, 기능/요구사항 정의서(현재 문서) 완성
    - 핵심 KPI(온보딩 완료율, 자산 연동 완료율 등) 정의
  2. 2단계 – MVP 와이어프레임 & 웹 프로토타입 구현
    - React 기반 MVP 와이어프레임 화면 구현 (홈/마켓/라이브/커뮤니티/마이, 로그인/가입)
    - 현재처럼 Nginx 정적 서빙 구조로 기획서와 1:1 대응되는 화면 구성
    - 더미 데이터로 자산/상품/라이브/커뮤니티 동작 흐름 검증
  3. 3단계 – 백엔드 API · DB 연동
    - FastAPI + MySQL 기반의 계정/프로필/콘텐츠 기본 모델 설계
    - 핵심 기능(회원가입, 로그인, 마켓 리스트, 커뮤니티 글 조회/등록) API 구현
    - docker-compose로 frontend · backend · db 통합 실행 환경 구축 및 테스트
  4. 4단계 – AI 인사이트 1차 적용 (요약/브리핑 중심)
    - 홈 화면: AI 자산 한 줄 요약 카드(FR-A01)에 더미 또는 샘플 API 연동
    - 마켓/라이브/커뮤니티: “AI에게 물어보기”, “핵심 요약 & Q&A 하이라이트” 버튼/영역에 실제 응답 연동
    - AI 응답 시간·품질에 대한 내부 지표 정의 및 기본 로깅
  5. 5단계 – 모바일 앱 전환 및 고도화
    - React Native 기반 모바일 앱으로 전환 (동일 IA/와이어프레임 재사용)
    - 푸시 알림, 실시간 라이브 스트리밍, 자산 연동 기능 고도화
    - 운영/지표 추적을 위한 관리자 도구(어드민 웹) 및 통계 화면 추가

9-3. 개발 · 테스트 · 운영 환경 전략

  • 개발 환경 – 로컬 Docker 기반 실행 (단일 docker-compose), 더미 데이터 사용
  • 테스트 환경 – 동일 docker-compose 설정을 클라우드 VM에 배포, 실제 DB 스키마 검증
  • 운영(향후) – 프론트/백엔드/DB를 개별 서비스로 분리 가능한 구조 전제
  • 환경별 설정은 .env 파일과 docker-compose override로 분리 관리

9-4. 리스크 및 대응 방향(요약)

  • AI 응답 지연/품질 리스크
    → 초기에는 더미/부분 기능만 노출, 핵심 플로우(로그인·탐색·커뮤니티)는 AI 없이도 완주 가능하도록 설계
  • DB 스키마 변경 리스크
    → MVP 단계에서는 단순 스키마로 시작 후, 마이그레이션 도구 도입 검토
  • 인프라 운영 리스크
    → docker-compose 기반 단일 서버 운영에서 시작, 트래픽 증가 시 컨테이너 단위 확장 및 클라우드 매니지드 서비스(RDS 등) 전환
