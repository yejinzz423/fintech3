요구사항 정의서(Requirement) 이론

요구사항 정의서는 서비스가 만족해야 하는 기능적·비기능적 조건을 명확히 명시하는 문서이다. 이 문서는 QA 테스트 기준, 개발 범위, 기능 단위 분할의 근거가 되며 “무엇을 만들 것인지”에 대한 공식 기준서 역할을 한다. EasyFin V2.0에서는 전통적인 FR/NFR뿐 아니라, AI 모델 품질, 설명 가능성, 데이터 보호까지 함께 요구사항으로 다룬다.

  • 기능 요구사항(Functional Requirements, FR)
  • 비기능 요구사항(Non-functional Requirements, NFR)
  • AI 관련 요구사항 (성능·지연·오류 처리·설명 가능성·데이터 활용 범위)
  • 제약조건(Constraints) 및 버전별 변경 이력 관리

본 문서는 서비스 목적 & 목표 → 페르소나 3종 → 문제 정의 → 가치 제안 → IA → 기능 정의 흐름에서 도출된 내용을 요구사항 수준으로 구체화한 결과이며, 이후 테스트 케이스, 개발 일정, 릴리즈 노트의 기준이 된다.

EasyFin V2.0 요구사항 정의서 (고도화 버전)

본 문서는 EasyFin V2.0 서비스의 기능 정의, 화면 정의, 사용자 흐름, 비기능 요구사항을 명확히 기술한 문서로, IA 정보구조도 문서 및 와이어프레임 문서와 함께 세트로 버전 관리된다. 특히 AI 기반 자산 분석/추천 기능이 포함되므로, 화면·플로우뿐 아니라 데이터 처리와 응답 품질에 대한 요구사항까지 포함한다.

1. 문서 목적

본 요구사항 정의서는 EasyFin V2.0의 MVP 및 이후 고도화 버전에 대해 개발·디자인·기획 단계에서 일관된 기준을 제공하기 위함이다. 특히 AI 기반 자산 분석/추천 기능 도입에 따라, 전통적인 화면·플로우 요구사항뿐 아니라 데이터 처리, 응답 시간, 설명 가능성, 개인정보 보호에도 명시적인 기준을 둔다.

2. 서비스 개요

  • 서비스명: EasyFin V2.0
  • 목적: 개인 투자자를 위한 금융정보 탐색·자산 관리·커뮤니티·라이브 기반 금융 서비스
  • 플랫폼: 모바일 앱(React Native + Expo) / 프로토타입(React Web + FastAPI 백엔드)

    • “AI가 먼저 읽고 정리해 주는 금융 인텔리전스” (Intelligence · Trust)

4. 전체 시스템 구성

EasyFin V2.0 MVP는 빠른 개발·테스트·배포를 목표로 컨테이너 기반 모듈형 아키텍처를 채택하였다. 각 기능은 독립된 Docker 컨테이너로 분리되어 있으며, 서비스 간 의존성과 배포 용이성을 고려한 3-Tier 구조(Frontend · Backend · Database)로 구성된다.

4.1 주요 구성 요소 (MVP 기준 기술 스택 및 아키텍처)

  • ① 전체 아키텍처 개요

    MVP는 아래 3개의 핵심 서비스로 구성되며 docker-compose를 통해 통합 관리한다.

    • Frontend: 정적 웹 UI (React로 제작된 HTML/CSS를 Nginx로 서빙)
    • Backend API: FastAPI 기반 REST API 컨테이너
    • Database: MySQL 8.0 기반의 상태 저장(Storage) 컨테이너

    각 서비스는 독립적으로 배포 및 재시작이 가능하며, 서비스 간 네트워크는 docker-compose 내부 브리지 네트워크를 통해 연결된다.

  • ② Database (MySQL 8.0)
    • 이미지: mysql:8.0 (컨테이너명: easyfin-mysql)
    • 데이터베이스명: easyfin_db
    • 계정 정책:
      • 사용자 계정: easyfin / easyfin
      • 루트 계정: rootpassword
    • 데이터 영속성: mysql_data:/var/lib/mysql 볼륨 사용
    • 시간대 설정: TZ=Asia/Seoul (로그 및 데이터 기록 일관성 유지 목적)
    • 헬스체크 구성:
      mysqladmin ping -h localhost -uroot -prootpassword
      → DB가 정상적으로 쿼리를 받을 수 있는지 확인 후 backend 기동
    • 보안 정책:
      • 외부 포트는 기본 비공개(3306 미노출)
      • 백엔드 컨테이너만 접근 가능하도록 내부 네트워크 격리
  • ③ Backend API (FastAPI + Gunicorn/Uvicorn)
    • 컨테이너 빌드:
      • 베이스 이미지: python:3.10-slim
      • requirements.txt 기반 패키지 설치
    • 애플리케이션 실행 구조:
      gunicorn main:app -k uvicorn.workers.UvicornWorker --bind 0.0.0.0:8000 --workers 1
      → worker 수를 1로 제한한 이유: 초기 DB 스키마 생성(create_all) 시 레이스 컨디션 방지
    • 환경 변수:
      • DATABASE_URL=mysql+pymysql://easyfin:easyfin@db:3306/easyfin_db
      • TZ=Asia/Seoul
    • 포트 매핑: 8000 → 8000
    • 기능 역할:
      • 사용자 인증/회원가입
      • 마켓 데이터 제공
      • 커뮤니티 API 제공
      • 라이브 목록 반환
      • 향후 AI 분석/요약 API 연동 포인트
    • 의존성 관리: depends_on: db(service_healthy) → DB 준비 후 API 실행
  • ④ Frontend (Nginx 정적 웹)
    • 베이스 이미지: nginx:alpine (경량 이미지로 배포 효율화)
    • 배포 구조:
      • index.html, styles.css만 제공하는 정적 SPA 형태
      • React 컴포넌트 기반 UI를 HTML로 컴파일한 MVP용 구조
    • 포트 매핑: 3000 → 80
    • 역할:
      • MVP 기준 홈/마켓/라이브/커뮤니티/마이 페이지 정적 제공
      • 향후 React Native 앱 전환 시 웹 와이어프레임 역할 수행
  • ⑤ 공통 운영 · DevOps 구성
    • 재기동 정책: 모든 서비스에 restart: unless-stopped 적용 → 예기치 않은 장애 발생 시 자동 복구
    • 서비스 간 통신: docker 내부 브리지 네트워크를 통한 서비스명 DNS 사용 예: backend → db 접속 시 db:3306
    • 보안 정책:
      • DB 외부 포트 미노출
      • 환경 변수 기반 민감 정보 관리
      • 추후 OAuth/KMS 연동 고려 가능
    • 확장성 전략(MVP → 정식 버전):
      • frontend → CDN/SPA 구조로 확장 가능
      • backend → Gunicorn worker 증가 및 컨테이너 수평 확장(HPA) 가능
      • DB → RDS 전환 및 읽기/쓰기 분리 전략 가능
      • AI 기능 → 별도 AI inference 서버/벡터DB로 분리 예정

8. 비기능 요구사항(NFR)

8.1 성능

  • 홈/마켓/라이브/커뮤니티 화면은 평균 1초 이내 최초 렌더링을 목표로 한다.
  • 네트워크 환경이 열악한 경우라도, 주요 정보(텍스트/리스트)는 3초 이내에 표시되도록 한다.

8.2 보안

  • 비밀번호는 백엔드에서 단방향 암호화하여 저장한다.
  • 회원 인증 이후 API 호출은 JWT 기반 인증을 사용한다(향후 단계 도입 가능).
  • 민감 정보(자산/계좌 번호 등)는 전송 및 저장 시 마스킹 또는 암호화 정책을 적용한다.

8.3 확장성

  • 상품 데이터, 라이브 목록은 API 기반 확장 가능한 구조로 설계한다.
  • 마켓 상세 정보는 차트·연결 콘텐츠(라이브/커뮤니티 글) 연계 기능을 추가할 수 있도록 확장성을 확보한다.
  • 초기에는 단일 리전/단일 인스턴스로 시작하되, 트래픽 증가 시 수평 확장이 가능한 구조를 고려한다.
